Introduction


Strings are one of the most common objects used in Java programming. They are among the only ways that programmers can show readable information to their consumers. Generally, a string will represent every word in every sentence, whether displayed to the user, or returned by the user. This means that, though a string is not one of the core Java primitives, it is still an extremely important part of Java as a whole, and is treated as such by language itself.
Here, we will investigate what a string actually is in Java, what makes it unique, and why programmers might use a few of the string alternatives.


Strings


Strings are a special type of Java object, but that doesn't mean that they aren't objects. In fact, Java Strings are included in the Java.lang package, and they extend the Object class. They also implement the marker interface Serializable, as well as the Comparable interface used for comparing objects. Finally, String implements the CharSequence interface which allows its implementing classes to represent a sequence of characters. StringBuilder and StringBuffer will also implement this interface.
It is also important to note that the String class is Final and Immutable.
Final: The String class cannot be extended.
Immutable: Strings cannot be changed once created.
Because Strings are immutable, a new String must be created every time programmers attempt to manipulate them. This means that even changing one letter creates a new String. Luckily, Strings have a special area in memory for their storage, unlike most objects.
This special place in memory is called the String Pool. Though it is on the heap, where all objects are stored, it behaves very differently. The String Pool contains only String Literals. Strings and String Literals aren't quite the same thing. As we've gone over, Strings are actually objects, not primitives. When a String Object is created, it is stored on the heap, as usual. String Literals are technically String Objects as well, but they stored in the String Pool to increase performance. String Literals are also represented by a character string such as "Hello" instead of the normal 'new String()' used for Objects.
The differences between the two are pretty confusing, but very important. This is because String Literals in the String Pool can be referenced in new String declarations, unlike the String Objects, which contain specific references to objects in the heap. In order to best exemplify this, look at the following example.

String a = "I am a String Literal";
String b = "I am a String Literal";
String c = new String("I am a String Object");
String d = new String("I am a String Object");

Because Strings a and b are both String Literals with the same value, they are actually referencing the same String Literal in the String Pool. However, because Strings c and d are String Objects, they are referencing two different Strings on the heap, even though they are the same. Thus, it is almost always better to declare a String using a String Literal than a String Object, to save space in memory.
Finally, Strings have the special ability to be combined together using the "+" operator, unlike most objects. Do note that combining a String in this way still creates a new String Literal, because Strings are immutable. For example:
String a = "Hello " + "World!";
creates a new String Literal that reads "Hello World!" as an addition of the two Strings.


StringBuilders


StringBuilders have very similar functionality to Strings, but they have a few important differences. Like Strings, they extend from Object and implement Serializable and CharSequence. Also like Strings, StringBuilders are simply sequences of characters.
However, unlike Strings, StringBuilders are mutable. This means that all StringBuilder character strings can be easily changed without having to create a new StringBuilder. As a result, StringBuilders are far better used when a programmer is planning on changing a string.
StringBuilders do have one obvious downside. They aren't Thread-Safe.
Thread-Safe: A piece of code is Thread-Safe when multiple threads can access it at once and still function correctly.
This means that, where a StringBuilder is quite fast, it sacrifices its ability to be effective in multithreaded enviroments.
It should also be noted that StringBuilders are best used when building or changing a string, not actually storing it. Strings have far more functionality than both StringBuilders and StringBuffers, as StringBuilders are mainly used for their fast character inserts, replacements, appends, and deletions.
StringBuilders are also stored with a specific capacity, which increases as the character array increases. Strings are often stored as String Literals, as we went over before, which make them more optimal for referencing and storing.


StringBuilders


StringBuffers are almost identical to StringBuilders in functionality. Like their counterparts, they extend from Object and implement Serializable and CharSequence. They are also mutable like StringBuilders.
The main, and really only, difference between the two occurs in their speed and Thread-Safety. StringBuffers are Thread-Safe, unlike StringBuilders, making them far more ideal to use in a multithreaded enviroment.
However, like each of these different classes, StringBuffers have their downside. They are a deal slower than Strings and StringBuilders. This means that using many StringBuffers can greatly slow down a program in comparison to a program using only Strings or StringBuilders.
